# 기능 정책 정리

## 문서 목적

이 문서는 `web-viewer`에서 제공할 기능들이 어떤 정책으로 동작해야 하는지 정리한 문서입니다.

여기서 말하는 정책은 다음을 의미합니다.

- 기능의 역할
- 입력과 출력
- 사용자에게 보여야 하는 동작
- 실패 시 처리 방식
- `web-viewer`와 `cpp-core` 중 누가 담당하는지

## 공통 정책

### 1. 역할 분리 정책

- `web-viewer`는 렌더링, UI, 사용자 인터랙션을 담당한다.
- `cpp-core`는 STL 분석, 메쉬 통계, 기하 계산을 담당한다.
- 같은 정보를 둘 다 계산할 수 있더라도, 최종적으로 어떤 레이어를 신뢰할지 정해야 한다.

### 2. 실패 처리 정책

- 파일 로드 실패는 사용자에게 바로 보이는 메시지로 안내한다.
- 파서 실패와 렌더링 실패는 구분해서 표시한다.
- `cpp-core` 분석 실패가 발생하더라도, 뷰어 렌더링 자체는 가능한 한 계속 동작한다.

### 3. 상태 표시 정책

- 현재 로딩 중인지, 성공했는지, 실패했는지를 상태 메시지로 보여준다.
- 메쉬 정보 패널에는 가능한 한 마지막 성공 결과를 유지한다.

### 4. 데이터 신뢰 정책

- 화면 렌더링에 필요한 geometry는 `web-viewer`가 직접 보유한다.
- STL 메타데이터는 현재 단계에서 `cpp-core` 결과를 우선 신뢰한다.
- 나중에 WebAssembly 전환 후에는 geometry 생성도 `cpp-core` 결과를 우선할 수 있다.

## 1. 파일 로딩 정책

### 목적

사용자가 STL 또는 OBJ 파일을 선택했을 때, 가능한 한 빠르게 로드하고 실패 시 원인을 알 수 있게 한다.

### 정책

- 지원 포맷은 `STL`, `OBJ`로 한정한다.
- `STL`은 ASCII와 Binary를 모두 지원한다.
- `OBJ`는 기본적인 vertex / face 구조를 우선 지원한다.
- 파일 확장자만 보지 말고, 실제 데이터 형식도 함께 고려한다.

### 사용자 경험

- 파일 선택 직후 로딩 상태가 표시되어야 한다.
- 성공 시 파일명, 포맷, triangle 수, bounds가 보인다.
- 실패 시 “왜 실패했는지” 알 수 있는 문구가 보여야 한다.

### 책임

- `web-viewer`: 파일 선택, 읽기, 기본 파싱, 상태 표시
- `cpp-core`: STL 메타데이터 검증 및 분석

## 2. 렌더링 정책

### 목적

로드한 메쉬를 사용자가 바로 볼 수 있도록 안정적으로 렌더링한다.

### 정책

- 렌더링은 항상 `Three.js BufferGeometry` 기준으로 처리한다.
- 로드 후 `computeVertexNormals()`를 호출해 기본 조명을 안정화한다.
- 기본 재질은 메쉬 형태를 잘 보여주는 무광 계열 재질을 사용한다.
- 초기에 `DoubleSide`를 허용해 면 방향 문제로 모델이 안 보이는 상황을 줄인다.

### 사용자 경험

- 로드하면 모델이 바로 화면 안에 들어와야 한다.
- 모델이 너무 작거나 너무 커서 보이지 않는 일이 없어야 한다.
- 별도 조작 없이도 대략적인 형태를 확인할 수 있어야 한다.

### 책임

- `web-viewer`

## 3. 카메라 정책

### 목적

사용자가 메쉬를 자연스럽게 회전, 이동, 확대/축소할 수 있게 한다.

### 정책

- 기본 조작은 `OrbitControls`를 사용한다.
- 모델 로드 시 bounding sphere를 기준으로 자동 프레이밍한다.
- 카메라 `near`, `far`는 모델 크기에 맞춰 재계산한다.
- 로드 후 카메라 타깃은 항상 모델 중심으로 맞춘다.

### 사용자 경험

- 처음 로드했을 때 모델이 시야 밖에 있으면 안 된다.
- 마우스 조작이 과도하게 빠르거나 느리지 않아야 한다.

### 책임

- `web-viewer`

## 4. Wireframe 정책

### 목적

사용자가 메쉬의 삼각형 구조를 직관적으로 확인할 수 있게 한다.

### 정책

- Wireframe은 토글 방식으로 제공한다.
- Solid와 Wireframe은 같은 geometry를 공유한다.
- Wireframe은 디버깅과 구조 분석 목적이므로 성능보다 가시성을 우선한다.

### 사용자 경험

- 버튼 또는 체크박스로 즉시 켜고 끌 수 있어야 한다.
- 상태 전환 시 화면 반응이 즉시 보여야 한다.

### 책임

- `web-viewer`

## 5. Bounding Box 정책

### 목적

사용자가 메쉬의 전체 범위와 크기를 이해할 수 있게 한다.

### 정책

- Bounding Box는 기본적으로 토글 가능해야 한다.
- 현재 단계에서는 `Three.js BoxHelper` 시각화를 사용한다.
- 표시되는 Bounds 수치는 가능한 경우 `cpp-core` 분석 결과를 우선 반영한다.

### 사용자 경험

- 메쉬의 대략적인 크기와 위치 범위를 쉽게 파악할 수 있어야 한다.

### 책임

- 시각화: `web-viewer`
- 수치 계산 우선값: `cpp-core`

## 6. Section Plane 정책

### 목적

사용자가 메쉬 내부를 잘라 보며 구조를 분석할 수 있게 한다.

### 정책

- MVP에서는 clipping plane 1개를 우선 지원한다.
- 축 고정 방식으로 시작하고, 나중에 축 전환을 추가한다.
- 슬라이더 값은 모델 크기와 무관하게 동작하되, 향후 bounds 기반 범위로 개선할 수 있다.

### 사용자 경험

- 슬라이더를 움직이면 절단 결과가 즉시 보여야 한다.
- 메쉬가 완전히 사라지는 경우에도 원인을 이해할 수 있어야 한다.

### 책임

- `web-viewer`

## 7. Picking 정책

### 목적

사용자가 메쉬의 특정 위치를 선택하고 분석할 수 있게 한다.

### 정책

- Picking은 `Raycaster` 기반으로 구현한다.
- 최소 단계에서는 “선택 지점 표시”까지만 우선 지원한다.
- 선택 결과는 시각적 피드백이 반드시 있어야 한다.

### 사용자 경험

- 메쉬를 클릭했을 때 사용자가 “선택되었다”는 사실을 즉시 알 수 있어야 한다.
- 선택이 실패하면 아무 반응이 없는 것보다 상태 메시지를 주는 편이 좋다.

### 책임

- 현재 단계: `web-viewer`
- 향후 BVH 최적화: `cpp-core`

## 8. 메쉬 정보 패널 정책

### 목적

사용자가 현재 로드된 모델의 핵심 정보를 빠르게 확인하게 한다.

### 정책

- 최소 표시 항목:
  - 파일명
  - 포맷
  - Vertex 수
  - Triangle 수
  - Bounds
  - 분석 소스
- 숫자는 읽기 쉬운 형식으로 표시한다.
- 가능하면 “이 값이 어디서 왔는지”를 드러낸다.

### 사용자 경험

- 로드 후 별도 클릭 없이 정보가 보인다.
- 값이 바뀌면 즉시 패널에 반영된다.

### 책임

- 표시: `web-viewer`
- STL 메타데이터 우선값: `cpp-core`

## 9. cpp-core 연동 정책

### 목적

웹 뷰어가 C++ 코어를 실제로 활용하고 있다는 점을 드러낸다.

### 정책

- 현재 단계에서는 `STL 메타데이터 분석`에 한정해 연결한다.
- 렌더링 실패와 `cpp-core` 분석 실패는 분리해서 처리한다.
- `cpp-core` 결과가 없더라도 뷰어 자체는 계속 사용할 수 있어야 한다.

### 사용자 경험

- STL 업로드 시 분석이 완료되었다는 상태가 보여야 한다.
- 분석이 실패해도 파일 로드나 렌더링까지 모두 막히지 않아야 한다.

### 책임

- `web-viewer`: 업로드, 요청, 결과 반영
- `cpp-core`: 분석 계산

## 10. FPS / 성능 표시 정책

### 목적

사용자가 뷰어 상태와 성능 수준을 인지할 수 있게 한다.

### 정책

- MVP에서는 간단한 FPS 표시만 우선 제공한다.
- FPS는 디버깅 목적이므로 지나치게 화려할 필요는 없다.
- 성능이 급격히 떨어질 때 원인 파악에 도움이 되도록 유지한다.

### 책임

- `web-viewer`

## 11. 이후 확장 정책

추가 기능은 아래 원칙을 따른다.

- 렌더링 기능보다 분석 기능 우선
- 의료용 3D Viewer 맥락을 해치지 않는 기능 우선
- `cpp-core`가 맡을 계산과 `web-viewer`가 맡을 시각화를 계속 분리

우선순위가 높은 확장:

- 거리 측정
- Picking 고도화
- BVH 기반 가속
- 다중 section plane

우선순위가 낮은 확장:

- 고급 조명
- 그림자
- deferred rendering
- 인스턴싱
- 복잡한 후처리
